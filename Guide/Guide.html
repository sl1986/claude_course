Claude Certified Architect – Foundations Certification Exam Guide

Introduction
Сертификация Claude Certified Architect – Foundations подтверждает, что специалисты способны принимать обоснованные решения о компромиссах при внедрении реальных решений с использованием Claude. Данный экзамен проверяет базовые знания по Claude Code, Claude Agent SDK, Claude API и Model Context Protocol (MCP) — ключевым технологиям, используемым для создания production‑уровня приложений с Claude.

Вопросы экзамена основаны на реалистичных сценариях, заимствованных из реальных клиентских кейсов, включая построение агентных систем для поддержки клиентов, проектирование многоагентных исследовательских пайплайнов, интеграцию Claude Code в CI/CD‑процессы, создание инструментов для повышения продуктивности разработчиков и извлечение структурированных данных из неструктурированных документов. Кандидаты должны продемонстрировать не только концептуальные знания, но и практическое понимание архитектуры, конфигурации и компромиссов при развёртывании решений в продакшене.

Данное руководство описывает содержание экзамена, перечисляет домены и проверяемые задачи, приводит примеры вопросов и рекомендует стратегии подготовки. Используйте его вместе с практическим опытом для эффективной подготовки.

Target Candidate Description
Идеальный кандидат для данной сертификации — архитектор решений, который проектирует и внедряет production‑приложения с использованием Claude. Такой кандидат обладает практическим опытом в следующих областях:

  • Построение агентных приложений с использованием Claude Agent SDK, включая многоагентную оркестрацию, делегирование подагентам, интеграцию инструментов и lifecycle‑хуки
  • Настройка и кастомизация Claude Code для командных рабочих процессов с использованием файлов CLAUDE.md, Agent Skills, интеграций MCP‑серверов и plan mode
  • Проектирование интерфейсов инструментов и ресурсов Model Context Protocol (MCP) для интеграции backend‑систем
  • Проектирование промптов, обеспечивающих надёжный структурированный вывод, с использованием JSON‑схем, few‑shot‑примеров и паттернов извлечения
  • Эффективное управление контекстным окном при работе с длинными документами, многошаговыми диалогами и передачей контекста между агентами
  • Интеграция Claude в CI/CD‑пайплайны для автоматизированного code review, генерации тестов и обратной связи по pull request’ам
  • Принятие обоснованных решений по эскалации и надёжности, включая обработку ошибок, human‑in‑the‑loop‑процессы и паттерны самооценки

Кандидат, как правило, имеет 6 и более месяцев практического опыта разработки с использованием Claude API, Agent SDK, Claude Code и MCP, понимая как возможности, так и ограничения больших языковых моделей в production‑средах.

Exam Content

Response Types
Все вопросы экзамена имеют формат multiple choice. Каждый вопрос содержит один правильный ответ и три неправильных варианта (дистрактора).

Выберите один ответ, который наилучшим образом завершает утверждение или отвечает на вопрос.

Дистракторы — это варианты ответа, которые может выбрать кандидат с неполными знаниями или опытом.

Невыбранные вопросы засчитываются как неправильные; штрафа за угадывание нет.

Exam Results
Экзамен имеет статус «сдан» или «не сдан». Оценивание проводится по минимальному стандарту, установленному экспертами предметной области.

Результаты сообщаются в виде шкалированной оценки от 100 до 1 000. Минимальный проходной балл — 720.

Шкалированная система оценки используется для выравнивания результатов между разными версиями экзамена, которые могут незначительно отличаться по уровню сложности.

Exam Scenarios

В экзамене используются вопросы, основанные на сценариях. Каждый сценарий представляет собой реалистичный производственный контекст, который задаёт рамки для набора вопросов. Во время экзамена будут представлены 4 сценария, случайным образом выбранные из полного набора 6 сценариев, приведённых ниже.

Scenario 1: Customer Support Resolution Agent
Вы разрабатываете агент для разрешения обращений в службе поддержки с использованием Claude Agent SDK. Агент обрабатывает запросы с высокой степенью неопределённости, такие как возвраты, споры по биллингу и проблемы с учётными записями. Он имеет доступ к backend‑системам через пользовательские инструменты Model Context Protocol (MCP) (get_customer, lookup_order, process_refund, escalate_to_human). Ваша цель — обеспечить 80%+ решений с первого контакта, при этом корректно определяя моменты, когда необходимо передать запрос человеку.

Основные домены: Agentic Architecture & Orchestration, Tool Design & MCP Integration, Context Management & Reliability

Scenario 2: Code Generation with Claude Code
Вы используете Claude Code для ускорения разработки программного обеспечения. Ваша команда применяет его для генерации кода, рефакторинга, отладки и создания документации. Необходимо интегрировать Claude Code в рабочий процесс разработки с использованием пользовательских slash‑команд, конфигураций CLAUDE.md, а также понимать, когда следует использовать plan mode, а когда — прямое выполнение.

Основные домены: Claude Code Configuration & Workflows, Context Management & Reliability

Scenario 3: Multi-Agent Research System
Вы разрабатываете многоагентную исследовательскую систему с использованием Claude Agent SDK. Координирующий агент делегирует задачи специализированным подагентам: один осуществляет поиск в интернете, один анализирует документы, один синтезирует результаты, а один генерирует отчёты. Система исследует темы и формирует комплексные отчёты с источниками.

Основные домены: Agentic Architecture & Orchestration, Tool Design & MCP Integration, Context Management & Reliability

Scenario 4: Developer Productivity with Claude
Вы создаёте инструменты для повышения продуктивности разработчиков с использованием Claude Agent SDK. Агент помогает инженерам изучать незнакомые кодовые базы, понимать устаревшие системы, генерировать шаблонный код и автоматизировать повторяющиеся задачи. Он использует встроенные инструменты (Read, Write, Bash, Grep, Glob) и интегрируется с MCP‑серверами.

Основные домены: Tool Design & MCP Integration, Claude Code Configuration & Workflows, Agentic Architecture & Orchestration

Scenario 5: Claude Code for Continuous Integration
Вы интегрируете Claude Code в пайплайн Continuous Integration / Continuous Deployment (CI/CD). Система выполняет автоматические code review, генерирует тест‑кейсы и предоставляет обратную связь по pull request’ам. Необходимо проектировать промпты, которые дают практически полезную обратную связь и минимизируют количество ложных срабатываний.

Основные домены: Claude Code Configuration & Workflows, Prompt Engineering & Structured Output

Scenario 6: Structured Data Extraction
Вы разрабатываете систему извлечения структурированных данных с использованием Claude. Система извлекает информацию из неструктурированных документов, валидирует результат с помощью схем JavaScript Object Notation (JSON) и поддерживает высокую точность. Она должна корректно обрабатывать пограничные случаи и интегрироваться с downstream‑системами.

Основные домены: Prompt Engineering & Structured Output, Context Management & Reliability

Domain 1: Agentic Architecture & Orchestration

Task Statement 1.1: Design and implement agentic loops for autonomous task execution

Knowledge of:

  • Жизненный цикл агентного цикла: отправка запросов Claude, анализ stop_reason
    ("tool_use" против "end_turn"), выполнение запрошенных инструментов и возврат
    результатов для следующей итерации
  • Как результаты выполнения инструментов добавляются в историю диалога, чтобы модель
    могла рассуждать о следующем действии
  • Различие между модель‑ориентированным принятием решений (Claude самостоятельно
    оценивает, какой инструмент вызвать на основе контекста) и заранее
    сконфигурированными деревьями решений или последовательностями инструментов

Skills in:

  • Реализация управляющей логики агентного цикла, который продолжается при
    stop_reason = "tool_use" и завершается при stop_reason = "end_turn"
  • Добавление результатов инструментов в контекст диалога между итерациями, чтобы модель
    могла использовать новую информацию в рассуждениях
  • Избежание антипаттернов, таких как разбор естественного языка для определения
    завершения цикла, использование произвольных лимитов итераций как основного
    механизма остановки или проверка текстового ответа ассистента как индикатора
    завершения

Task Statement 1.2: Orchestrate multi-agent systems with coordinator–subagent patterns

Knowledge of:

  • Архитектура «hub‑and‑spoke», в которой координирующий агент управляет всей
    межагентной коммуникацией, обработкой ошибок и маршрутизацией информации
  • То, как подагенты работают с изолированным контекстом — они не наследуют историю
    диалога координатора автоматически
  • Роль координатора в декомпозиции задач, делегировании, агрегации результатов и выборе
    подагентов в зависимости от сложности запроса
  • Риски чрезмерно узкой декомпозиции задач координатором, приводящие к неполному
    охвату широких исследовательских тем

Skills in:

  • Проектирование координаторов, которые анализируют требования запроса и динамически
    выбирают, каких подагентов вызывать, вместо обязательного прохождения всей цепочки
  • Разделение области исследования между подагентами для минимизации дублирования
    (например, назначение каждому агенту отдельных подтем или типов источников)
  • Реализация итеративных циклов уточнения, в которых координатор оценивает синтез,
    выявляет пробелы, повторно делегирует поиск и анализ с уточнёнными запросами и
    повторяет синтез до достижения полноты
  • Маршрутизация всей межагентной коммуникации через координатора для наблюдаемости,
    унифицированной обработки ошибок и контролируемого потока информации

Task Statement 1.3: Configure subagent invocation, context passing, and spawning

Knowledge of:

  • Инструмент Task как механизм создания подагентов и требование, чтобы
    allowedTools включал "Task" для возможности их вызова координатором
  • То, что контекст подагента должен передаваться явно в промпте — подагенты не наследуют
    контекст родителя автоматически и не разделяют память между вызовами
  • Конфигурация AgentDefinition, включая описания, системные промпты и ограничения
    по инструментам для каждого типа подагентов
  • Управление сессиями на основе fork‑подхода для исследования альтернативных путей
    от общего аналитического базиса

Skills in:

  • Прямое включение полных результатов предыдущих агентов в промпт подагента
    (например, передача результатов веб‑поиска и анализа документов агенту синтеза)
  • Использование структурированных форматов данных для разделения контента и
    метаданных (URL‑источников, названий документов, номеров страниц) при передаче
    контекста между агентами для сохранения атрибуции
  • Создание параллельных подагентов путём генерации нескольких вызовов инструмента
    Task в одном ответе координатора, а не в отдельных ходах
  • Проектирование промптов координатора, которые задают цели исследования и критерии
    качества вместо пошаговых процедурных инструкций, обеспечивая адаптивность подагентов

Task Statement 1.4: Implement multi-step workflows with enforcement and handoff patterns

Knowledge of:

  • Различие между программным принуждением (хуки, предварительные проверки) и
    промпт‑ориентированным управлением порядком выполнения
  • Ситуации, где требуется детерминированное соблюдение требований
    (например, проверка личности перед финансовыми операциями), поскольку одних
    промптов недостаточно и они имеют ненулевую вероятность отказа
  • Структурированные протоколы передачи задач при эскалации в середине процесса,
    включающие данные клиента, анализ первопричин и рекомендуемые действия

Skills in:

  • Реализация программных предварительных условий, блокирующих последующие вызовы
    инструментов до завершения необходимых шагов
    (например, блокировка process_refund до получения подтверждённого customer ID
    через get_customer)
  • Декомпозиция сложных клиентских запросов на отдельные элементы с последующим
    параллельным расследованием каждого в рамках общего контекста и финальным
    синтезом единого решения
  • Формирование структурированных сводок передачи
    (customer ID, первопричина, сумма возврата, рекомендуемое действие) при эскалации к
    человеческим агентам, не имеющим доступа к полному диалогу

Task Statement 1.5: Apply Agent SDK hooks for tool call interception and data normalization

Knowledge of:

  • Шаблоны хуков (например, PostToolUse), перехватывающие результаты инструментов

  • Шаблоны хуков, перехватывающие исходящие вызовы инструментов для обеспечения
    соблюдения правил (например, блокировка возвратов выше лимита)
  • Различие между использованием хуков для детерминированных гарантий и применением
    промптов для вероятностного соблюдения правил

Skills in:

  • Реализация хуков PostToolUse для нормализации разнородных форматов данных
    (unix‑время, ISO‑8601, числовые статус‑коды) из различных MCP‑инструментов
  • Реализация хуков перехвата вызовов, блокирующих действия, нарушающие политику
    (например, возвраты свыше $500), с перенаправлением в альтернативные процессы
    (например, эскалация к человеку)
  • Выбор хуков вместо промпт‑ориентированного управления, когда бизнес‑правила требуют
    гарантированного соблюдения

Knowledge of:

  • Паттерны prompt chaining, разбивающие проверки на последовательные этапы

  • Ценность адаптивных планов исследования, которые формируют подзадачи на основе

Skills in:

  • Выбор подходящего паттерна декомпозиции в зависимости от характера задачи:
    prompt chaining для предсказуемых многоаспектных проверок и динамическая
    декомпозиция для открытых исследовательских задач
  • Разделение крупных code review на локальный анализ каждого файла с отдельным этапом

  • Декомпозиция открытых задач

    структуры, определения зон наибольшего влияния и формирования приоритетного плана,
    который адаптируется по мере выявления зависимостей

Task Statement 1.7: Manage session state, resumption, and forking

Knowledge of:

  • fork_session для создания независимых веток от общего аналитического состояния для

  • Важность информирования агента об изменениях в ранее проанализированных файлах при

  • Почему начало новой сессии со структурированной сводкой надёжнее, чем возобновление

Skills in:

  • Использование fork_session для параллельного анализа альтернатив

  • Выбор между возобновлением сессии (когда предыдущий контекст в основном актуален) и

  • Информирование возобновлённой сессии о конкретных изменениях файлов для целевого
    повторного анализа, вместо полной переоценки

Domain 2: Tool Design & MCP Integration

Task Statement 2.1: Design effective tool interfaces with clear descriptions and boundaries

Knowledge of:

  • Описания инструментов как основной механизм выбора инструментов LLM; краткие или
    расплывчатые описания приводят к ненадёжному выбору между похожими инструментами
  • Важность включения форматов входных данных, примеров запросов, пограничных случаев и описаний границ применения в описание инструмента
  • Как неоднозначные или пересекающиеся описания инструментов приводят к неправильной маршрутизации (например, analyze_content и analyze_document с почти идентичными описаниями)
  • Влияние формулировок системного промпта на выбор инструментов: инструкции, чувствительные к ключевым словам, могут создавать непреднамеренные ассоциации

Skills in:

  • Написание описаний инструментов, которые чётко разграничивают назначение каждого
    инструмента, ожидаемые входные и выходные данные и условия использования по сравнению с альтернативами
  • Переименование инструментов и обновление описаний для устранения функционального
    перекрытия (например, переименование analyze_content в extract_web_results с веб‑ориентированным описанием)
  • Разделение универсальных инструментов на специализированные с чёткими контрактами
    вход/выход (например, разделение analyze_document на extract_data_points, summarize_content и verify_claim_against_source)
  • Проверка системных промптов на наличие инструкций, чувствительных к ключевым словам,
    которые могут переопределять хорошо написанные описания инструментов

Scenario: Customer Support Resolution Agent

Question 1:
Производственные данные показывают, что в 12% случаев ваш агент полностью пропускает вызов get_customer и вызывает lookup_order, используя только имя клиента, указанное пользователем. В отдельных случаях это приводит к неверной идентификации аккаунтов и ошибочным возвратам средств. Какое изменение наиболее эффективно решит эту проблему надёжности?

A) Добавить программное предварительное условие, которое блокирует вызовы lookup_order и process_refund до тех пор, пока get_customer не вернёт подтверждённый идентификатор клиента.
B) Улучшить системный промпт, указав, что верификация клиента через get_customer обязательна перед любыми операциями с заказами.
C) Добавить few-shot‑примеры, демонстрирующие, что агент всегда сначала вызывает get_customer, даже если клиент сам сообщает детали заказа.
D) Реализовать классификатор маршрутизации, который анализирует каждый запрос и включает только подмножество инструментов, подходящих для данного типа запроса.

Correct Answer: A

Когда для критически важной бизнес‑логики требуется строгая последовательность вызова инструментов (например, проверка личности клиента перед возвратом средств), программное принуждение обеспечивает детерминированные гарантии, которые невозможно получить с помощью промптов. Варианты B и C полагаются на вероятностное соблюдение инструкций LLM, что недопустимо, когда ошибки имеют финансовые последствия. Вариант D решает проблему доступности инструментов, а не их порядка, что не является корнем проблемы.

Question 2:
Производственные логи показывают, что агент часто вызывает get_customer, когда пользователи спрашивают о заказах (например, «проверь мой заказ №12345»), вместо вызова lookup_order. Оба инструмента имеют минимальные описания («Получает информацию о клиенте» / «Получает детали заказа») и принимают схожие форматы идентификаторов. Какой первый шаг наиболее эффективно повысит надёжность выбора инструмента?

A) Добавить few-shot‑примеры в системный промпт, демонстрирующие корректные паттерны выбора инструментов, с 5–8 примерами маршрутизации запросов о заказах в lookup_order.
B) Расширить описание каждого инструмента, добавив форматы входных данных, примеры запросов, пограничные случаи и чёткие границы применения по сравнению с похожими инструментами.
C) Реализовать слой маршрутизации, который анализирует пользовательский ввод перед каждым ходом и заранее выбирает инструмент на основе ключевых слов и паттернов идентификаторов.
D) Объединить оба инструмента в единый lookup_entity, принимающий любой идентификатор и самостоятельно определяющий, к какому backend обращаться.

Correct Answer: B

Описания инструментов — основной механизм, с помощью которого LLM выбирают инструменты. Когда описания минимальны, у модели нет достаточного контекста для различения похожих инструментов. Вариант B напрямую устраняет первопричину с минимальными затратами и высокой эффективностью. Few-shot‑примеры (A) увеличивают расход токенов, не устраняя корневую проблему. Слой маршрутизации (C) избыточен и обходит нативное понимание языка моделью. Объединение инструментов (D) — допустимое архитектурное решение, но требует значительно больше усилий, чем уместно для «первого шага».

Ваша команда хочет снизить стоимость API для автоматизированного анализа. В настоящее время вызовы Claude в реальном времени используются для двух рабочих процессов:
(1) блокирующая pre‑merge‑проверка, которая должна завершиться до того, как разработчики смогут выполнить merge, и
(2) отчёт по техническому долгу, который формируется ночью и просматривается на следующее утро.

Ваш менеджер предлагает перевести оба процесса на использование Message Batches API ради 50% экономии стоимости. Как следует оценить это предложение?

A) Использовать пакетную обработку только для отчётов по техническому долгу; оставить вызовы в реальном времени для pre‑merge‑проверок.
B) Перевести оба процесса на пакетную обработку с опросом статуса для определения завершения.
C) Оставить вызовы в реальном времени для обоих процессов, чтобы избежать проблем с порядком результатов batch‑обработки.
D) Перевести оба процесса на пакетную обработку с резервным переходом на реальное время при превышении таймаута.

Correct Answer: A

Вариант D добавляет лишнюю сложность, когда более простое решение — сопоставить API с соответствующим сценарием использования.

Question 12:
Pull request изменяет 14 файлов в модуле отслеживания складских запасов. Однопроходное ревью, анализирующее все файлы одновременно, даёт непоследовательные результаты: подробная обратная связь по некоторым файлам и поверхностные комментарии по другим, пропущенные очевидные баги, противоречивые замечания — один и тот же паттерн помечается как проблемный в одном файле и одобряется в другом в рамках одного PR. Как следует перестроить процесс ревью?

A) Разделить ревью на сфокусированные проходы: сначала анализировать каждый файл отдельно на локальные проблемы, затем выполнить отдельный интеграционный проход для анализа межфайловых потоков данных.
B) Обязать разработчиков разбивать крупные PR на более мелкие (по 3–4 файла) перед запуском автоматизированного ревью.
C) Перейти на модель более высокого уровня с большим контекстным окном для анализа всех 14 файлов за один проход.
D) Запускать три независимых прохода ревью всего PR и отмечать только те проблемы, которые повторяются минимум в двух из трёх запусков.

Correct Answer: A

Разделение ревью на сфокусированные проходы напрямую устраняет первопричину — расфокусировку внимания при обработке большого количества файлов одновременно. Анализ файл‑за‑файлом обеспечивает равномерную глубину, а отдельный интеграционный проход позволяет выявлять межфайловые проблемы.
Вариант B перекладывает нагрузку на разработчиков, не улучшая систему.
Вариант C ошибочно предполагает, что увеличение окна контекста решает проблему качества внимания.
Вариант D, напротив, подавляет выявление реальных багов, требуя консенсуса для проблем, которые могут проявляться непредсказуемо.

Technologies and Concepts

Следующий перечень содержит технологии и концепции, которые могут встретиться на экзамене:

  • Claude Agent SDK — определения агентов, агентные циклы, обработка stop_reason, хуки (PostToolUse, перехват вызовов инструментов), создание подагентов через инструмент Task, конфигурация allowedTools
  • Model Context Protocol (MCP) — MCP‑серверы, MCP‑инструменты, MCP‑ресурсы, флаг isError, описания инструментов, распределение инструментов, конфигурация .mcp.json, подстановка переменных окружения
  • Claude Code — иерархия CLAUDE.md (пользователь / проект / каталог), .claude/rules/ с YAML‑path‑scoping, .claude/commands/ для slash‑команд, .claude/skills/ с фронтматтером SKILL.md (context: fork, allowed-tools, argument-hint), plan mode, прямое выполнение, команды /memory, /compact, --resume, fork_session, подагент Explore
  • Claude Code CLI — флаг -p / --print для неинтерактивного режима, --output-format json, --json-schema для структурированного CI‑вывода
  • Claude API — tool_use с JSON‑схемами, варианты tool_choice ("auto", "any", принудительный выбор инструмента), значения stop_reason, max_tokens, системные промпты
  • Message Batches API — 50% экономии стоимости, до 24 часов обработки, custom_id, опрос статуса, отсутствие поддержки многошаговых tool‑вызовов
  • JSON Schema — обязательные и необязательные поля, enum‑типы, nullable‑поля, паттерн "other" + детализация, строгий режим для исключения синтаксических ошибок
  • Pydantic — валидация схем, семантические ошибки, циклы валидации‑повтора
  • Встроенные инструменты — Read, Write, Edit, Bash, Grep, Glob — назначение и критерии выбора
  • Few‑shot prompting — целевые примеры для неоднозначных сценариев, демонстрация формата, обобщение
  • Prompt chaining — последовательная декомпозиция задач
  • Управление контекстным окном — бюджеты токенов, прогрессивная суммаризация, эффект «потери в середине», извлечение контекста, scratchpad‑файлы
  • Управление сессиями — возобновление, fork_session, именованные сессии, изоляция контекста
  • Оценка уверенности — уверенность на уровне полей, калибровка по размеченным наборам, стратифицированная выборка
