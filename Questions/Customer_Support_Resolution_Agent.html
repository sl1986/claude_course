Customer Support Resolution Agent — Training Questions

You are building a customer support resolution agent using the Claude Agent SDK. The agent handles high-ambiguity requests like returns, billing disputes, and account issues. It has access to your backend systems through custom Model Context Protocol (MCP) tools (get_customer, lookup_order, process_refund, escalate_to_human). Your target is 80%+ first-contact resolution while knowing when to escalate.

Вы создаёте агента по разрешению вопросов службы поддержки клиентов с использованием Claude Agent SDK. Агент обрабатывает запросы с высокой неоднозначностью, такие как возвраты, споры по выставлению счетов и проблемы с аккаунтом. Он имеет доступ к вашим бэкэнд-системам через пользовательские инструменты MCP (get_customer, lookup_order, process_refund, escalate_to_human). Ваша цель — более 80% решений с первого обращения, при этом зная, когда эскалировать.

Question 1

Your get_customer tool returns all matches when searching by name. Claude currently picks the customer with the most recent order when multiple results are returned, but production data shows this causes 15% of multi-match cases to proceed with the wrong customer account. How should you address this?

Ваш инструмент get_customer возвращает все совпадения при поиске по имени. Клод сейчас выбирает клиента с самым последним заказом, когда возвращается несколько результатов, но производственные данные показывают, что это приводит к неправильному аккаунту в 15% случаев с несколькими совпадениями. Как следует это исправить?

A: Modify get_customer to return only the single most likely match based on a ranking algorithm, simplifying Claude's decision by eliminating ambiguous results.
A: Измените get_customer так, чтобы он возвращал только одно наиболее вероятное совпадение на основе алгоритма ранжирования, упрощая решение Клода за счёт устранения неоднозначных результатов.
B: Add few-shot examples showing Claude how to use conversational context (products mentioned, dates referenced) to infer the correct customer without requiring clarification.
B: Добавьте несколько примеров, показывающих Клоду, как использовать разговорный контекст (упомянутые продукты, даты), чтобы определить правильного клиента без необходимости уточнения.
C: Implement a confidence scoring system that proceeds automatically above 85% confidence and prompts for clarification below that threshold.
C: Реализуйте систему оценки уверенности, которая автоматически продолжает работу при уверенности выше 85% и запрашивает уточнение ниже этого порога.
D: Instruct Claude to ask for an additional identifier (email, phone, or order number) when get_customer returns multiple matches, before taking any customer-specific action.
D: Инструктируйте Клода спросить дополнительный идентификатор (email, телефон или номер заказа), когда get_customer возвращает несколько совпадений, прежде чем предпринимать какие-либо действия, привязанные к клиенту.
Explanation

Asking the user for an additional identifier such as email, phone, or order number is the most reliable way to disambiguate multiple matches. One extra conversational turn is a small cost compared to the 15% error rate caused by selecting the wrong account.

Запрос у пользователя дополнительного идентификатора, такого как email, телефон или номер заказа, является самым надёжным способом устранить неоднозначность при нескольких совпадениях. Один дополнительный ход разговора — это небольшая цена по сравнению с 15% ошибкой при выборе неверного аккаунта.

Question 2

Production logs reveal that the agent misinterprets data from your MCP tools: Unix timestamps from get_customer, ISO 8601 dates from lookup_order, and numeric status codes (1=pending, 2=shipped). Some tools are third-party MCP servers you cannot modify. What's the most maintainable approach to normalize data formats?

Производственные журналы показывают, что агент неправильно интерпретирует данные из ваших инструментов MCP: Unix-временную метку из get_customer, ISO 8601 даты из lookup_order и числовые коды статуса (1=ожидание, 2=отправлено). Некоторые инструменты — это сторонние MCP-серверы, которые вы не можете изменить. Какой наиболее поддерживаемый подход для нормализации форматов данных?

A: Create a normalize_data tool that the agent calls after each data retrieval to transform values.
A: Создайте normalize_data-инструмент, который агент вызывает после каждого извлечения данных для преобразования значений.
B: Add detailed format documentation to your system prompt explaining each tool's data conventions.
B: Добавьте подробную документацию по форматам в системный промпт, объясняющую соглашения по данным каждого инструмента.
C: Modify tools you control to return human-readable formats; create wrapper tools for third-party tools.
C: Измените инструменты, которыми вы управляете, чтобы они возвращали человекочитаемые форматы; создайте обёртки для сторонних инструментов.
D: Use a PostToolUse hook to intercept tool results and apply formatting transformations before agent processing.
D: Используйте хук PostToolUse, чтобы перехватывать результаты инструментов и применять преобразования формата до обработки агентом.
Explanation

A PostToolUse hook gives you a centralized, deterministic point to normalize all tool outputs before the agent sees them, including third-party MCP tool results. This is more maintainable than relying on the agent or tool callers to remember explicit normalization steps.

Хук PostToolUse предоставляет централизованную, детерминированную точку для нормализации всех результатов инструментов до того, как агент их обработает, включая результаты сторонних MCP-инструментов. Это более поддерживаемый подход, чем полагаться на то, что агент или вызывающий инструмент запомнит явные шаги нормализации.

Логи показывают, что он эскалирует простые случаи (стандартные замены при повреждениях с фотодоказательствами), но пытается самостоятельно решать сложные ситуации, требующие исключений из политики. Какой наиболее эффективный способ улучшить калибровку эскалации?

A: Deploy a separate classifier model trained on historical tickets to predict which requests need escalation before the main agent begins processing.
A: Разверните отдельную модель-классификатор, обученную на исторических обращениях, чтобы предсказывать, какие запросы требуют эскалации до того, как основной агент начнёт обработку.
B: Add explicit escalation criteria to your system prompt with few-shot examples demonstrating when to escalate versus resolve autonomously.
B: Добавьте явные критерии эскалации в системный промпт с редкими примерами, демонстрирующими, когда эскалировать, а когда решать самостоятельно.
C: Have the agent self-report a confidence score (1-10) before each response and automatically route requests to humans when confidence falls below a threshold.
C: Пусть агент сам сообщает оценку уверенности (1-10) перед каждым ответом и автоматически направляет запросы к людям, когда уверенность ниже порога.
D: Implement sentiment analysis to detect customer frustration levels and automatically escalate when negative sentiment exceeds a threshold.
D: Реализуйте анализ тональности, чтобы выявлять уровень неудовольствия клиента и автоматически эскалировать, когда негативная тональность превышает порог.
Explanation

Explicit escalation criteria with few-shot examples address the root cause: unclear decision boundaries between simple cases and policy-exception cases. This is a proportional first fix that teaches the agent when to escalate versus when to resolve autonomously.

Явные критерии эскалации с редкими примерами решают основную причину проблемы: неясные границы принятия решений между простыми случаями и случаями исключений из политики. Это пропорциональное первое исправление, которое учит агента, когда эскалировать, а когда решать самостоятельно.

Question 4

Production logs show the agent frequently calls get_customer when users ask about orders (e.g., "check my order #12345"), instead of calling lookup_order. Both tools have minimal descriptions ("Retrieves customer information" / "Retrieves order details") and accept similar identifier formats. What's the most effective first step to improve tool selection reliability?

Производственные журналы показывают, что агент часто вызывает get_customer, когда пользователи спрашивают про заказы (например, «проверьте мой заказ #12345»), вместо вызова lookup_order. Оба инструмента имеют минимальные описания («Получает информацию о клиенте» / «Получает информацию о заказе») и принимают похожие форматы идентификаторов. Какой наиболее эффективный первый шаг для повышения надежности выбора инструмента?

A: Implement a routing layer that parses user input before each turn and pre-selects the appropriate tool based on detected keywords and identifier patterns.
A: Реализуйте слой маршрутизации, который анализирует ввод пользователя перед каждым ходом и предварительно выбирает подходящий инструмент на основе обнаруженных ключевых слов и шаблонов идентификаторов.
B: Add few-shot examples to the system prompt demonstrating correct tool selection patterns, with 5-8 examples showing order-related queries routing to lookup_order.
B: Добавьте несколько примеров в системный промпт, показывающих правильные шаблоны выбора инструментов, с 5-8 примерами, демонстрирующими перенаправление запросов по заказам на lookup_order.
D: Expand each tool's description to include input formats it handles, example queries, edge cases, and boundaries explaining when to use it versus similar tools.
D: Расширьте описание каждого инструмента, чтобы включить форматы ввода, примеры запросов, пограничные случаи и границы использования по сравнению с похожими инструментами.
C: Consolidate both tools into a single lookup_entity tool that accepts any identifier and internally determines which backend to query.
C: Консолидируйте оба инструмента в один lookup_entity инструмент, который принимает любой идентификатор и внутренне определяет, к какому бэкенду обращаться.
Explanation

Expanding tool descriptions to include formats, example queries, edge cases, and explicit boundaries directly improves the agent's natural mechanism for tool selection. This is a low-effort, high-leverage first step that fixes the root cause of insufficient description-based guidance.

Расширение описаний инструментов, чтобы включить форматы ввода, примеры запросов, пограничные случаи и явные границы, непосредственно улучшает естественный механизм выбора инструментов агентом. Это недорогое и высокоэффективное первое действие, которое исправляет коренную причину недостаточной информации в описаниях.

В тестировании вы замечаете, что агент часто вызывает get_customer, когда пользователи спрашивают о статусе заказа, хотя lookup_order было бы более уместно. Что стоит рассмотреть в первую очередь, чтобы решить эту проблему?

A: Прочитайте описания инструментов, чтобы убедиться, что они чётко различают назначение каждого инструмента.

B: Добавьте несколько примеров, охватывающих все возможные шаблоны запросов, связанных с заказами, в системный промпт.

C: Реализуйте классификатор предварительной обработки, который обнаруживает запросы о заказах и направляет их напрямую на lookup_order.

После вызовов get_customer и lookup_order агент получил все доступные системные данные, но столкнулся с неопределённостью. Какая ситуация является наиболее подходящим триггером для вызова escalate_to_human?

A: В сообщении клиента упоминается как вопрос по выставлению счетов, так и возврат товара. Агент должен эскалировать, чтобы человек мог координировать решение обеих вопросов в одном взаимодействии.

Пробел в политике по ценам конкурентов — это настоящий триггер для эскалации, поскольку агент не может безопасно решить его без человеческого суждения. Другие сценарии обычно можно обработать с помощью существующих процедур и общения с клиентом.

B: Реализуйте классификатор маршрутизации, который анализирует каждый запрос и активирует только тот поднабор инструментов, который подходит для этого типа запроса.

Программная предварительная проверка устраняет неоднозначность, обеспечивая требуемую последовательность в коде: lookup_order и process_refund не могут выполняться, пока не будет подтверждён идентификатор клиента. Это надёжнее, чем только инструкции в промпте.

Какова наиболее вероятная причина этого расхождения?

A: Системный промпт содержит чувствительные к ключевым словам инструкции, которые направляют поведение на основе терминов вроде «аккаунт», создавая непреднамеренные шаблоны выбора инструмента.

B: Описания инструментов нуждаются в дополнительных негативных примерах, указывающих, когда НЕ использовать каждый инструмент, чтобы предотвратить эту путаницу, вызванную ключевыми словами.

Систематическая закономерность, зависящая от ключевого слова, указывает на то, что инструкции в промпте направляют поведение на основе термина «аккаунт». Поскольку описания инструментов уже однозначны, наиболее вероятная причина расхождения — явные инструкции в промпте, а не неоднозначность описаний.
