Multi‑Agent Research System — Training Set

You are building a multi-agent research system using the Claude Agent SDK. A coordinator agent delegates to specialized subagents: one searches the web, one analyzes documents, one synthesizes findings, and one generates reports. The system researches topics and produces comprehensive, cited reports.

Вы создаёте многоагентную исследовательскую систему с использованием CDK Claude Agent. Координатор-агент делегирует задачи специализированным субагентам: один ищет в интернете, другой анализирует документы, третий синтезирует находки, третий генерирует отчёты. Система исследует темы и готовит комплексные, цитируемые отчёты.

Question 1

Production monitoring reveals inconsistent synthesis quality. When aggregated results reach ~75K tokens, the synthesis agent reliably uses information from the beginning and end of the input but frequently omits critical findings from the middle. How should you restructure the aggregated input?

Производственный мониторинг выявляет нестабильное качество синтеза. Когда агрегированные результаты достигают ~75 000 токенов, агент стабильно использует данные из начала и конца, но часто упускает важную информацию из середины. Как следует реструктурировать агрегированный вход?

A: Rotate which subagent’s results appear first across tasks
A: Чередовать порядок результатов субагентов
B: Place a key findings summary at the beginning and organize results with clear section headers
B: Разместить сводку ключевых выводов в начале и структурировать данные с помощью заголовков
C: Stream subagent outputs incrementally
C: Передавать результаты субагентов поэтапно
D: Aggressively summarize all outputs under 20K tokens
D: Сжать все результаты до менее 20 000 токенов
Explanation

Positioning key findings at the start leverages the primacy effect, while explicit section headers improve navigation and mitigate the “lost in the middle” attention problem.

Размещение ключевых выводов в начале использует эффект приоритета, а заголовки разделов помогают избежать эффекта «потерянности посередине».

Question 3

During a materials research task, the web search subagent queries three source categories with different outcomes: academic databases returned 15 relevant papers, industry reports returned "0 results found," and patent databases returned "Connection timeout." When designing error propagation to the coordinator, what approach enables the best recovery decisions?

Во время задачи по исследованию материалов субагент веб-поиска обращается к трём категориям источников с разными результатами: академические базы данных возвращают 15 релевантных статей, отраслевые отчёты — «0 найденных результатов», а патентные базы данных — «тайм-аут соединения». При проектировании передачи ошибок координатору, какой подход позволяет принимать наилучшие решения по восстановлению?

A: Aggregate outcomes into a single success rate metric (e.g., "67% source coverage") with detailed logs available on request.
A: Агрегировать результаты в единую метрику успеха (например, «67% охват источника») с подробными журналами, доступными по запросу.
B: Distinguish access failures (timeout) needing retry decisions from valid empty results ("0 results") representing successful queries.
B: Различать сбои доступа (тайм-аут), требующие решения о повторной попытке, и валидные пустые результаты («0 результатов»), представляющие успешные запросы.
C: Report both the timeout and "0 results" as failures requiring coordinator intervention.
C: Сообщайте о тайм-ауте и о «0 результатах» как о отказах, требующих вмешательства координатора.
D: Have the subagent retry transient failures internally and only report persistent errors.
D: Пусть субагент повторно обрабатывает временные ошибки самостоятельно и сообщает только о стойких ошибках.
Explanation

Distinguishing a timeout from a valid empty result preserves actionable detail: the coordinator can retry the timed-out patent database while accepting the industry report query as a meaningful empty finding.

Различение тайм-аута и валидного пустого результата сохраняет полезные детали: координатор может повторно запросить патентную базу данных и одновременно принять запрос отраслевого отчёта как информативный пустой результат.

Субагент анализа документов успешно обработал все предоставленные документы. Субагент синтеза теперь должен получить сводку результатов из этого смешанного качества. A: Proceed with synthesis using only the successful sources, generating output without indicating which data was unavailable.
A: Продолжить синтез, используя только успешные источники, генерируя вывод без указания, какие данные были недоступны.

C: Пусть субагент синтеза попросит координатора повторно попробовать тайм-аут источники с увеличенными тайм-аутами перед началом работы, обеспечив полное покрытие данных до начала синтеза.

D: Структурировать вывод синтеза с аннотациями охвата, указывая, какие выводы хорошо подтверждены, а какие темы имеют пробелы из-за недоступных источников.
Explanation

Structuring output with coverage annotations preserves useful findings while transparently showing where gaps exist, enabling informed confidence judgments without discarding completed work.

Структурирование вывода с аннотациями охвата сохраняет полезные результаты и при этом прозрачно показывает, где есть пробелы, что позволяет делать обоснованные выводы о степени доверия без отказа от уже выполненной работы.

Question 5

The document analysis subagent frequently encounters failures when processing PDF files—some have corrupted sections causing parsing exceptions, others are password-protected, and occasionally the parsing library times out on large files. Currently, any exception immediately terminates the subagent and returns an error to the coordinator, which must decide whether to retry, skip the document, or fail the entire research task. This is causing excessive coordinator involvement in routine error handling. What's the most effective architectural improvement?

Субагент анализа документов часто сталкивается с ошибками при обработке PDF-файлов — некоторые содержат повреждённые участки, вызывающие исключения парсинга, другие защищены паролем, а иногда библиотека парсинга истекает по тайм-ауту на больших файлах. В настоящее время любое исключение немедленно завершает работу субагента и возвращает ошибку координатору, который должен решить, повторить ли попытку, пропустить документ или завершить всю задачу исследования. Это вызывает чрезмерное участие координатора в рутинной обработке ошибок. Какое архитектурное улучшение является наиболее эффективным?

A: Create a dedicated error-handling agent that monitors all subagent failures via a shared queue and makes recovery decisions independently, dispatching retry commands directly to subagents.
A: Создайте выделенный агент для обработки ошибок, который отслеживает все сбои субагентов через общую очередь и самостоятельно принимает решения о восстановлении, отправляя команды повторной попытки непосредственно субагентам.
B: Configure the subagent to always return partial results with success status, embedding error details in metadata. The coordinator treats all responses as successful and filters problematic items during synthesis.
B: Настройте субагент так, чтобы он всегда возвращал частичные результаты со статусом успеха, встраивая детали ошибок в метаданные. Координатор рассматривает все ответы как успешные и фильтрует проблемные элементы во время синтеза.
C: Have the coordinator validate all documents before dispatching to the subagent, rejecting documents likely to cause failures to ensure the subagent only receives processable files.
C: Пусть координатор проверяет все документы до отправки субагенту, отклоняя документы, которые, вероятно, вызовут сбои, чтобы субагент получал только обрабатываемые файлы.
D: Have the subagent implement local recovery for transient failures and only propagate errors it cannot resolve to the coordinator, including what was attempted and any partial results obtained.
D: Пусть субагент реализует локальное восстановление для временных сбоев и передаёт координатору только те ошибки, которые он не может разрешить, включая информацию о предпринятых попытках и полученные частичные результаты.
Explanation

Implementing local recovery for transient failures within the subagent reduces excessive coordinator involvement while still escalating truly unresolvable issues with full context and partial results.

Реализация локального восстановления временных сбоев внутри субагента снижает чрезмерное участие координатора, при этом обостряя действительно неразрешимые проблемы с полным контекстом и частичными результатами.

A: Пусть координатор явно разделит исследовательское пространство перед делегированием, назначая каждому агенту отдельные подтемы или типы источников

B: Реализовать механизм общего состояния, где агенты фиксируют текущую область фокуса, позволяя другим агентам динамически избегать дублирования выполняемой работы

Having the coordinator explicitly partition the research space prevents duplication at the source, preserving parallel efficiency while ensuring distinct coverage for each agent.

Явное разделение исследовательского пространства координатором предотвращает дублирование на корню, сохраняя эффективность параллельной работы и обеспечивая различное покрытие каждым агентом.

Агент анализа документов должен сохранить оба противоречивых показателя и отметить несоответствие с указанием источников, позволяя координатору или стадии синтеза сопоставить данные с учётом более широкого контекста.

Question 10

Ваша оценка показывает, что 85% этих проверок — это простые проверки фактов (даты, имена, статистика), а 15% требуют более глубокого изучения. A: Дать синтезирующему агенту ограниченный инструмент verify_fact для простых проверок, в то время как сложные верификации продолжают делегироваться веб-поисковому агенту через координатора.

B: Дать синтезирующему агенту доступ ко всем инструментам веб-поиска, чтобы он мог решать любые задачи верификации напрямую без круговых обходов через координатора.

C: Пусть веб-поисковый агент заранее кэширует дополнительный контекст вокруг каждого источника во время первоначального исследования, предугадывая, что может потребоваться синтезирующему агенту для проверки.
D: Have the synthesis agent accumulate all verification needs and return them as a batch to the coordinator at the end of its pass, which then sends them all to the web search agent at once.
D: Пусть синтезирующий агент аккумулирует все запросы на проверку и возвращает их пакетно координатору в конце своего прохода, который затем отправляет их все веб-поисковому агенту сразу.
